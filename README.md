# 🚀 DropVault Backend

DropVault Backend is a security-focused, modular backend system built with Flask. It is designed not just to handle authentication and sessions, but to act as a controllable and observable system where traffic, identity, and behavior can be monitored and regulated in real time.

Unlike traditional CRUD-style backends, DropVault emphasizes control layers such as atomic rate limiting, session fingerprinting, and defensive system design. The goal is not only to make the system “work”, but to make it resilient, adaptable, and predictable under stress or abuse.

---

## 📌 Overview

At its core, DropVault Backend is structured around the idea that no single layer should be trusted completely. Every component — from request handling to session storage — is built with the assumption that failure, abuse, or unexpected behavior can occur at any time.

The system is designed with the following principles:

* Security-first: Every feature is implemented with abuse scenarios in mind
* Stateless request handling with stateful tracking (sessions stored externally)
* Atomic operations for critical logic (rate limiting via Lua)
* Fail-safe design: system continues operating even when dependencies fail
* Clear separation of concerns for maintainability and scalability

## ⚙️ Current Features

### 🔐 Authentication System

The authentication system is built using a classic email and password flow, but with production-level considerations:

* Credentials are validated through a service layer (`services.auth.login`)
* Login requests are validated to prevent empty or malformed inputs
* Upon successful login, a secure session is created using a cryptographically strong session ID generated by `secrets.token_urlsafe`
* Authentication is session-based (server-side), avoiding reliance on client-stored tokens

Additionally, the system captures contextual metadata at login time, including:

* Client IP address (with proxy support)
* Browser information
* Operating system

This data forms the foundation for future anomaly detection and security enforcement.

---

### 🧠 Session Management

Session handling is implemented through a custom abstraction (`set_session`) rather than relying purely on Flask defaults. This allows greater flexibility and future extensibility.

Each session stores:

* `user_id`: unique identifier of the authenticated user
* `session_id`: securely generated session token
* `ip_address`: origin IP of the session
* `browser`: parsed browser name
* `os`: parsed operating system
* `created_at`: session creation timestamp (UTC)
* `last_active`: last activity timestamp
* `expires_at`: expiration timestamp

Design considerations:

* Sessions are server-side and can be extended to Redis or database storage
* Session lifecycle is explicitly tracked, enabling expiration enforcement
* Structure supports future features such as:

  * Multi-device login tracking
  * Manual session invalidation
  * Concurrent session limits

---

### 🌐 Network Tracking

The system includes a utility for extracting the real client IP address, even when behind reverse proxies such as Cloudflare.

It checks headers in order:

* `CF-Connecting-IP`
* `X-Forwarded-For`
* `remote_addr`

This ensures maximum accuracy in identifying the true origin of a request.

User-Agent strings are parsed using the `user-agents` library to extract:

* Browser family (e.g., Chrome, Firefox)
* Operating system (e.g., Windows, Android)

This information is used for lightweight device fingerprinting and can later support anomaly detection.

---

### 🚦 Rate Limiting (Advanced)

One of the core features of the system is its rate limiting mechanism, designed for correctness and scalability.

Key characteristics:

* Algorithm: Sliding Window using Redis Sorted Sets (ZSET)
* Execution: Atomic via Lua script (ensures no race conditions)
* Scope: Applied per IP and per endpoint
* Storage: Redis (compatible with Upstash)

Flow of execution:

1. Remove expired request entries from the sorted set
2. Count current requests within the time window
3. If the limit is exceeded, reject immediately
4. Otherwise, add the new request with a timestamp
5. Set expiration on the key

Atomicity is guaranteed because all steps run inside a single Lua script execution on Redis.

Fail-safe behavior:

* If Redis is unavailable or an error occurs, the system falls back to a fail-open strategy (requests are allowed rather than blocked)

Response headers included:

* `X-RateLimit-Limit`: maximum allowed requests
* `X-RateLimit-Remaining`: remaining requests in current window
* `X-RateLimit-Reset`: timestamp when limit resets

This allows frontend or clients to adapt behavior dynamically.

## 🧩 Architecture

The project follows a modular structure to ensure maintainability and scalability:

* `controllers/`: Handles HTTP requests and responses (Flask routes)
* `services/`: Contains business logic (authentication, session handling)
* `middlewares/`: Cross-cutting concerns (rate limiting, future auth guards)
* `utils/`: Helper functions (network utilities, parsing)
* `extensions/`: External integrations (Redis client, etc.)

This separation ensures:

* Easier testing and debugging
* Clear responsibility boundaries
* Ability to scale individual layers independently

## 🛠️ Tech Stack

* Backend Framework: Flask (Python)
* Data Store: Upstash Redis (used for rate limiting and session support)
* Security Tools:

  * `secrets` for cryptographic randomness
  * `user-agents` for parsing client device data
* Architecture Style: Modular, service-oriented backend

## 🔒 Security Highlights

The system incorporates several defensive design choices:

* Atomic rate limiting eliminates race conditions under high concurrency
* Session fingerprinting ties sessions to device context
* No reliance on client-side trust (all critical logic handled server-side)
* Fail-safe mechanisms prevent total system failure when dependencies break

These features ensure the system behaves predictably even under abnormal conditions.

## 🚀 Getting Started

To run the project locally:

1. Clone the repository:
   git clone <repo-url>
   cd dropvault-backend

2. Install dependencies:
   pip install -r requirements.txt

3. Configure environment variables:
   REDIS_URL=rediss://<your-upstash-url>
   FLASK_ENV=development
   SECRET_KEY=your_secret_key

4. Run the application:
   flask run

The server will start locally and be ready to accept requests.
